Sprint Goal: Before any employee can be added, the system must know the company structure, who has what access, and what an employee record looks like. These three modules are the absolute backbone. Nothing else works without them.

MODULE 1   |   Organization Setup
Purpose: Define the "shape" of the company inside the system. Every employee must belong to a department, hold a designation, and report to someone. Build this first.
1A — Departments
Departments are the top-level divisions of the company (HR, IT, Finance, Operations, etc.). Build a simple CRUD system for departments. Every employee will be linked to a department, so this must exist before the Employee module.
🗃  Table: hrms_departments
id              INT  PRIMARY KEY AUTO_INCREMENT
name            VARCHAR(100)  NOT NULL
code            VARCHAR(20)   NULLABLE  (optional short code e.g. HR, IT)
status          ENUM('active','inactive')  DEFAULT 'active'
created_at      TIMESTAMP  DEFAULT CURRENT_TIMESTAMP
updated_at      TIMESTAMP  DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
APIs to build:
GET    /departments         — list all departments (with status filter)
POST   /departments         — create new department
GET    /departments/:id     — single department detail
PUT    /departments/:id     — update department
DELETE /departments/:id     — soft delete (set status inactive, do not hard delete)
1B — Designations
Designations are job titles that belong to departments (e.g., 'Software Engineer' inside 'IT'). Always link designations to a department. When showing designation dropdowns, filter by selected department.
🗃  Table: hrms_designations
id              INT  PRIMARY KEY AUTO_INCREMENT
name            VARCHAR(100)  NOT NULL
department_id   INT  FK → hrms_departments(id)
level           VARCHAR(50)  NULLABLE  (e.g. Junior, Senior, Lead)
status          ENUM('active','inactive')  DEFAULT 'active'
created_at      TIMESTAMP  DEFAULT CURRENT_TIMESTAMP
updated_at      TIMESTAMP  DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
APIs to build:
GET    /designations              — list all (support ?department_id= filter)
POST   /designations              — create designation (department_id required)
GET    /designations/:id          — detail
PUT    /designations/:id          — update
DELETE /designations/:id          — soft delete
1C — Reporting Hierarchy
This is handled through the Employee table itself via a self-referencing manager_id field. You will build this when creating the Employee module, but the concept must be in your mind from day 1 because approval flows (leave, payroll) depend on it.
Key rule: When displaying the manager dropdown in the employee form, only show active employees with manager-eligible designations (or just all active employees — keep it simple first).
⚙  Development Steps — Follow This Order
1.  Create hrms_departments table and run migrations
2.  Build Departments CRUD APIs (all 5 endpoints)
3.  Test all department endpoints via Postman/API client
4.  Create hrms_designations table with FK to departments
5.  Build Designations CRUD APIs with department_id filter
6.  Test designation endpoints — verify department filter works
7.  Connect department + designation dropdowns in UI
8.  Verify soft delete works (does not hard delete, sets status=inactive)
⚠  Common Mistakes — Avoid These
✗  Do NOT hard delete departments or designations — employees are linked to them. Always soft delete.
✗  Do NOT allow designation creation without a department_id.
✗  Do NOT load all designations globally in one dropdown — always filter by department.
MODULE 2   |   Role & Permission (RBAC)
Purpose: Control who can see what, and who can do what. Implement this now. If you skip it and add it later, you will refactor every single module. The roles are already seeded in the DB. You need to build the permission mapping and route protection.
Roles Already in DB (Do Not Recreate)
→  Admin — Full system access
→  HR — Employee, payroll, attendance, leave management
→  Employee — Self-service portal only (ESS)
2A — Permissions
Define permissions as string slugs that map to module actions. Name them using module.action format.
Permission Slug
What It Allows
employee.create
Create a new employee record
employee.view
View employee profiles
employee.update
Edit employee information
employee.delete
Deactivate an employee
payroll.run
Execute payroll for a period
payroll.view
View payroll records and payslips
attendance.mark
Mark or manage attendance
attendance.view
View attendance reports
leave.request
Submit a leave request
leave.approve
Approve or reject leave requests
recruitment.manage
Create jobs and manage candidates
reports.view
Access reports and analytics
self.profile.view
View own profile (ESS)
self.payslip.view
View own payslip (ESS)
self.leave.request
Request own leave (ESS)
🗃  Table: hrms_permissions
id              INT  PRIMARY KEY AUTO_INCREMENT
name            VARCHAR(100)  UNIQUE  NOT NULL  (e.g. 'employee.create')
module          VARCHAR(50)   NOT NULL           (e.g. 'employee')
description     VARCHAR(255)  NULLABLE
created_at      TIMESTAMP  DEFAULT CURRENT_TIMESTAMP
🗃  Table: hrms_role_permissions
id              INT  PRIMARY KEY AUTO_INCREMENT
role_id         INT  FK → roles(id)             (existing roles table)
permission_id   INT  FK → hrms_permissions(id)
UNIQUE KEY      (role_id, permission_id)
2B — API / Route Protection
Every protected API must check: (1) Is the user logged in? (2) Does their role have the required permission? Implement a middleware that does this check. Apply it globally, not on each route individually.
Middleware logic:
Extract user from JWT token → get their role_id
Fetch permissions for that role from hrms_role_permissions
Check if required permission exists in their list
If yes → proceed. If no → return 403 Forbidden
Role-Permission mapping (seed this data):
Role
Permissions
Admin
ALL permissions
HR
employee.*, attendance.*, leave.approve, payroll.*, recruitment.*, reports.view
Employee
self.profile.view, self.payslip.view, self.leave.request, attendance.mark
⚙  Development Steps — Follow This Order
1.  Create hrms_permissions table and seed all permission slugs
2.  Create hrms_role_permissions table
3.  Seed role-permission mappings (Admin=all, HR=management, Employee=self-service)
4.  Build permission check middleware (JWT → role → permissions → allow/deny)
5.  Apply middleware to all existing and future routes
6.  Build /permissions endpoint for Admin to view/manage permissions
7.  Build /roles/:id/permissions endpoint to assign/revoke permissions from a role
8.  Test with each role — verify Admin can do everything, Employee is blocked from HR routes
⚠  Common Mistakes — Avoid These
✗  Do NOT check permissions inside business logic — always use middleware.
✗  Do NOT hardcode role names in conditions (e.g., if role === 'admin'). Always check permissions.
✗  Do NOT skip testing each role's access boundaries — one mistake here breaks security everywhere.
MODULE 3   |   Employee Management
Purpose: This is the heart of the entire system. Every other module (attendance, payroll, leave, performance) references an employee. Build this carefully and completely. Take your time here.
3A — Employee Profile (CRUD)
Build the complete employee record. Split data into logical groups to keep the UI clean.
🗃  Table: hrms_employees
id                  INT  PRIMARY KEY AUTO_INCREMENT
employee_code       VARCHAR(20)   UNIQUE  NOT NULL  (auto-generate: EMP-0001, EMP-0002)
first_name          VARCHAR(100)  NOT NULL
last_name           VARCHAR(100)  NOT NULL
email               VARCHAR(150)  UNIQUE  NOT NULL
phone               VARCHAR(20)   NULLABLE
cnic                VARCHAR(20)   NULLABLE
date_of_birth       DATE          NULLABLE
gender              ENUM('male','female','other')  NULLABLE
address             TEXT          NULLABLE
profile_photo       VARCHAR(255)  NULLABLE  (file path)
department_id       INT  FK → hrms_departments(id)
designation_id      INT  FK → hrms_designations(id)
manager_id          INT  FK → hrms_employees(id)  NULLABLE  (self-reference)
employment_type     ENUM('full_time','part_time','contract','intern')  DEFAULT 'full_time'
joining_date        DATE  NOT NULL
status              ENUM('active','inactive','suspended','resigned')  DEFAULT 'active'
user_id             INT  FK → users(id)  NULLABLE  (system login account)
created_at          TIMESTAMP  DEFAULT CURRENT_TIMESTAMP
updated_at          TIMESTAMP  DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
APIs to build:
GET    /employees               — list with filters (department, status, search by name/code)
POST   /employees               — create employee (auto-generate employee_code)
GET    /employees/:id           — full profile including department, designation, manager name
PUT    /employees/:id           — update employee
PATCH  /employees/:id/status    — change status (active/inactive/suspended/resigned)
3B — Document Upload
🗃  Table: hrms_employee_documents
id              INT  PRIMARY KEY AUTO_INCREMENT
employee_id     INT  FK → hrms_employees(id)
document_type   VARCHAR(100)  (e.g. CNIC, Contract, Certificate, Experience Letter)
file_name       VARCHAR(255)  NOT NULL
file_path       VARCHAR(500)  NOT NULL
uploaded_by     INT  FK → users(id)
created_at      TIMESTAMP  DEFAULT CURRENT_TIMESTAMP
APIs to build:
POST   /employees/:id/documents         — upload document (multipart form)
GET    /employees/:id/documents         — list all documents for employee
GET    /employees/:id/documents/:docId  — download/view single document
DELETE /employees/:id/documents/:docId  — delete document
File storage rules:
Store files in a dedicated folder: /storage/employees/{employee_id}/documents/
Save only the file path in DB, never the binary content
Validate file type (PDF, JPG, PNG only) and max size (5MB) on server side
3C — Status Management
Employee status controls system access and payroll eligibility. When status changes, related effects must happen automatically.
Status
Effect
active
Normal access. Included in payroll. Can login.
inactive
No login access. Excluded from payroll runs. Records preserved.
suspended
Blocked from login. HR can still manage record. Payroll paused.
resigned
Permanently excluded from payroll. Historical data preserved. Login blocked.
Important: When setting status to inactive/suspended/resigned, also disable the linked user account (set users.is_active = 0). Do this in the same transaction.
⚙  Development Steps — Follow This Order
1.  Create hrms_employees table with all fields — run migration
2.  Build employee code auto-generation logic (EMP-0001 format)
3.  Build POST /employees API with department + designation + manager linking
4.  Build GET /employees with pagination, search, and department filter
5.  Build GET /employees/:id that returns full profile (join department, designation, manager name)
6.  Build PUT /employees/:id for updates
7.  Build PATCH /employees/:id/status with user account sync logic
8.  Create hrms_employee_documents table
9.  Build document upload API with file validation and path storage
10.  Build document list and download APIs
11.  Build employee list UI with search + filters
12.  Build employee profile page with tabs: Profile | Job Info | Documents
13.  Build employee create/edit form with dependent dropdowns (department → designation)
14.  Test manager dropdown (self-reference — employee should NOT appear in their own manager list)
⚠  Common Mistakes — Avoid These
✗  Never hard delete an employee record — always soft delete via status change.
✗  Employee code must be unique and auto-generated — never let the user type it manually.
✗  Department and designation must be validated together — a designation from Department A cannot be assigned when Department B is selected.
✗  Always exclude the current employee from the manager dropdown (cannot manage themselves).
✗  Status change must be atomic — user account update and employee status update in one transaction.

MODULE 4   |   Attendance System
Purpose: Track when each employee starts and ends their workday. Calculate work hours. This data feeds directly into payroll (deductions for late/absent) and reporting.
4A — Check-in / Check-out
🗃  Table: hrms_attendance
id                  INT  PRIMARY KEY AUTO_INCREMENT
employee_id         INT  FK → hrms_employees(id)
date                DATE  NOT NULL
check_in_time       TIME  NULLABLE
check_out_time      TIME  NULLABLE
total_hours         DECIMAL(5,2)  NULLABLE  (auto-calculated on check-out)
status              ENUM('present','absent','late','half_day','on_leave')  NOT NULL
check_in_note       TEXT  NULLABLE
check_out_note      TEXT  NULLABLE
marked_by           INT  FK → users(id)  NULLABLE  (if HR marks it manually)
created_at          TIMESTAMP  DEFAULT CURRENT_TIMESTAMP
updated_at          TIMESTAMP  DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
UNIQUE KEY          (employee_id, date)   ← prevent duplicate records per day
APIs to build:
POST   /attendance/check-in             — mark check-in for today (one per employee per day)
POST   /attendance/check-out            — mark check-out, auto-calculate total_hours
GET    /attendance                      — list with filters (employee, date, month, status)
GET    /attendance/today                — today's attendance for all employees
GET    /attendance/employee/:id         — full attendance history for one employee
POST   /attendance/manual               — HR manually adds/corrects attendance record
Work hours calculation logic:
total_hours = TIMEDIFF(check_out_time, check_in_time) converted to decimal hours
If total_hours < 4 → mark as half_day. If no check-out by end of day → mark as present with note.
Define late threshold (e.g., 9:30 AM) — if check_in_time > threshold → status = late
4B — Attendance Logs & Reports
Build query APIs that HR and managers can use to filter and review attendance. These will also be used by the reporting module in Sprint 4.
Monthly attendance summary per employee (days present, absent, late, on_leave)
Department-wise attendance for a given date
Late arrivals list for a period
Missing check-out report (employee checked in but no check-out recorded)
⚙  Development Steps — Follow This Order
1.  Create hrms_attendance table with UNIQUE KEY on (employee_id, date)
2.  Build check-in API — prevent duplicate check-in for same day
3.  Build check-out API — calculate and store total_hours
4.  Build late detection logic using configurable threshold time
5.  Build manual attendance entry API for HR
6.  Build attendance list API with filters
7.  Build monthly summary aggregation query
8.  Build attendance UI — daily view and monthly calendar view per employee
9.  Build HR attendance dashboard showing today's present/absent/late counts
⚠  Common Mistakes — Avoid These
✗  Never allow two check-in records for the same employee on the same date.
✗  Never calculate total_hours in real-time on every request — calculate once on check-out and store it.
✗  Do NOT mix attendance logic with leave logic in the same service — keep them separate.
MODULE 5   |   Leave Management
Purpose: Employees request time off, managers/HR approve or reject. The system tracks remaining leave balance per employee per type. Approved leaves update the attendance record automatically.
5A — Leave Types Setup
🗃  Table: hrms_leave_types
id              INT  PRIMARY KEY AUTO_INCREMENT
name            VARCHAR(100)  NOT NULL  (Casual, Sick, Annual, Emergency)
max_days        INT  NOT NULL  (maximum allowed per year)
is_paid         BOOLEAN  DEFAULT TRUE
status          ENUM('active','inactive')  DEFAULT 'active'
🗃  Table: hrms_leave_balances
id              INT  PRIMARY KEY AUTO_INCREMENT
employee_id     INT  FK → hrms_employees(id)
leave_type_id   INT  FK → hrms_leave_types(id)
year            INT  NOT NULL  (e.g. 2025)
total_days      INT  NOT NULL  (copied from leave_types.max_days at year start)
used_days       INT  DEFAULT 0
remaining_days  INT  GENERATED ALWAYS AS (total_days - used_days) STORED
UNIQUE KEY      (employee_id, leave_type_id, year)
5B — Leave Requests & Approval
🗃  Table: hrms_leave_requests
id              INT  PRIMARY KEY AUTO_INCREMENT
employee_id     INT  FK → hrms_employees(id)
leave_type_id   INT  FK → hrms_leave_types(id)
start_date      DATE  NOT NULL
end_date        DATE  NOT NULL
total_days      INT  NOT NULL  (calculate: end_date - start_date + 1)
reason          TEXT  NULLABLE
status          ENUM('pending','approved','rejected','cancelled')  DEFAULT 'pending'
approved_by     INT  FK → users(id)  NULLABLE
approval_note   TEXT  NULLABLE
approved_at     TIMESTAMP  NULLABLE
created_at      TIMESTAMP  DEFAULT CURRENT_TIMESTAMP
Approval workflow:
Employee submits request → status = pending
Manager/HR sees pending requests → can approve or reject with note
On APPROVE: deduct from hrms_leave_balances (used_days += total_days), update attendance records for those dates to status = on_leave
On REJECT: no balance change. Notify employee.
On CANCEL (by employee before approval): status = cancelled, no balance impact
APIs to build:
POST   /leave/request                    — employee submits leave request
GET    /leave/requests                   — list requests (filter: status, employee, date)
GET    /leave/requests/pending           — pending approvals for manager/HR
PATCH  /leave/requests/:id/approve       — approve (HR/manager only — RBAC)
PATCH  /leave/requests/:id/reject        — reject with note (HR/manager only — RBAC)
PATCH  /leave/requests/:id/cancel        — cancel own pending request (employee only)
GET    /leave/balance/:employee_id       — view leave balance per type for current year
⚙  Development Steps — Follow This Order
1.  Create hrms_leave_types table and seed default types (Casual 10d, Sick 8d, Annual 14d, Emergency 3d)
2.  Create hrms_leave_balances table
3.  Build balance initialization logic — when a new employee joins, create balance records for all leave types for current year
4.  Create hrms_leave_requests table
5.  Build POST /leave/request with balance validation (reject if remaining_days < requested days)
6.  Build pending approvals list API
7.  Build approve API — atomic: update request status + deduct balance + update attendance records
8.  Build reject API with approval note
9.  Build cancel API (only for pending requests by own employee)
10.  Build leave balance view API
11.  Build leave request form UI
12.  Build HR approval dashboard (pending requests queue)
13.  Build employee leave history + balance view
⚠  Common Mistakes — Avoid These
✗  Always validate leave balance BEFORE allowing a request — reject if insufficient days.
✗  Approve and balance deduction must happen in one atomic transaction. If one fails, roll back both.
✗  When leave is approved, immediately update hrms_attendance for those dates to on_leave status.
✗  Do not let an employee cancel an already approved leave — only pending requests can be cancelled.

MODULE 7   |   Onboarding Flow
Purpose: Convert a selected candidate directly into an employee. This is the bridge between Recruitment and Employee Management. Pre-fill the employee form from candidate data to avoid duplicate entry.
7A — Candidate to Employee Conversion
When a candidate's application stage is set to 'selected', HR should see a 'Convert to Employee' button. This action pre-fills the employee creation form with data already available from the candidate.
Fields auto-filled from candidate:
name, email, phone from hrms_candidates
department_id from hrms_jobs.department_id
HR still manually sets: designation, manager, joining_date, employment_type
API:
POST   /recruitment/applications/:id/convert   — creates hrms_employees record, returns new employee_id
Immediately assign a system user account and the Employee role on conversion
7B — Onboarding Checklist
🗃  Table: hrms_onboarding_tasks
id              INT  PRIMARY KEY AUTO_INCREMENT
name            VARCHAR(200)  NOT NULL  (e.g. 'Upload CNIC', 'Assign system access')
is_default      BOOLEAN  DEFAULT TRUE  (default tasks shown for all new employees)
🗃  Table: hrms_employee_onboarding
id              INT  PRIMARY KEY AUTO_INCREMENT
employee_id     INT  FK → hrms_employees(id)
task_id         INT  FK → hrms_onboarding_tasks(id)
status          ENUM('pending','completed')  DEFAULT 'pending'
completed_by    INT  FK → users(id)  NULLABLE
completed_at    TIMESTAMP  NULLABLE
⚙  Development Steps — Follow This Order
1.  Build candidate-to-employee conversion API (POST /convert)
2.  Add user account creation and role assignment within conversion transaction
3.  Seed hrms_onboarding_tasks with default tasks
4.  Auto-create onboarding checklist for employee on conversion
5.  Build onboarding checklist view and task completion API
6.  Add 'Convert to Employee' button on application detail page (visible only for selected stage)
7.  Test full flow: Job → Application → Interview → Select → Convert → Employee appears in system
⚠  Common Mistakes — Avoid These
✗  Conversion must be all-or-nothing (atomic) — employee record + user account + role + onboarding checklist in one transaction.
✗  Check for duplicate email before converting — candidate email may already exist as a user.
✗  Do NOT allow converting the same application twice — check if employee already exists for that candidate.

MODULES 8 & 9   |   Payroll Setup & Processing
Critical note: Payroll is the most sensitive module in the system. Never recalculate old payroll. Always run payroll → store results → generate payslip. Payroll data must be immutable once processed.
8A — Salary Structure Setup
🗃  Table: hrms_salary_structures
id              INT  PRIMARY KEY AUTO_INCREMENT
name            VARCHAR(100)  NOT NULL  (e.g. 'Standard', 'Senior Staff', 'Intern')
status          ENUM('active','inactive')  DEFAULT 'active'
created_at      TIMESTAMP  DEFAULT CURRENT_TIMESTAMP
🗃  Table: hrms_salary_components
id              INT  PRIMARY KEY AUTO_INCREMENT
structure_id    INT  FK → hrms_salary_structures(id)
name            VARCHAR(100)  NOT NULL  (e.g. Basic Salary, House Allowance, Medical)
type            ENUM('earning','deduction')  NOT NULL
calc_type       ENUM('fixed','percentage')  NOT NULL
amount          DECIMAL(12,2)  NULLABLE  (if calc_type = fixed)
percentage      DECIMAL(5,2)  NULLABLE   (if calc_type = percentage, of basic salary)
is_taxable      BOOLEAN  DEFAULT FALSE
🗃  Table: hrms_employee_salaries
id              INT  PRIMARY KEY AUTO_INCREMENT
employee_id     INT  FK → hrms_employees(id)  UNIQUE  (one active salary per employee)
structure_id    INT  FK → hrms_salary_structures(id)
basic_salary    DECIMAL(12,2)  NOT NULL  (the base amount — percentages calculate from this)
effective_from  DATE  NOT NULL
created_at      TIMESTAMP  DEFAULT CURRENT_TIMESTAMP
9A — Payroll Processing
🗃  Table: hrms_payroll_runs
id              INT  PRIMARY KEY AUTO_INCREMENT
month           INT  NOT NULL  (1-12)
year            INT  NOT NULL
status          ENUM('draft','processing','completed','finalized')  DEFAULT 'draft'
processed_by    INT  FK → users(id)
processed_at    TIMESTAMP  NULLABLE

Payroll calculation logic (run per employee per month):
1. Fetch employee salary structure + basic_salary
2. Fetch attendance summary for the month (present, absent, late, leave days)
3. Calculate working days for the month (exclude weekends if applicable)
4. Per-day salary = basic_salary / working_days
5. Deduction for absent days = absent_days × per_day_salary
6. Apply each salary component (fixed amount OR percentage of basic)
7. net_salary = total_earnings − total_deductions
8. Store all component breakdowns in hrms_payroll_record_components
APIs to build:
POST   /salary-structures           — create structure
POST   /salary-structures/:id/components  — add component to structure
POST   /employees/:id/salary        — assign salary structure to employee
POST   /payroll/run                 — initiate payroll for month/year (processes all active employees)
GET    /payroll/runs                — list all payroll runs
GET    /payroll/runs/:id/records    — all employee records for a run
GET    /payroll/records/:id/payslip — generate/view payslip for one employee
PATCH  /payroll/runs/:id/finalize   — lock payroll (no changes after this)

Unified DB has shared tables — naming prevents conflicts
Never hard delete any record

RBAC middleware on every route

Store files as paths in DB
Never store binary file content in the database
Audit log critical actions
Creates accountability and helps with debugging production issues
Ask when requirements are unclear
Lead is always available — a wrong assumption wastes days of work
Requirements will change
Build APIs flexibly — avoid hardcoding business rules in multiple places
